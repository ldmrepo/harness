# 에이전트 설계 패턴 6가지

Phase 2에서 로드. 아키텍처 패턴 선택 시 참조한다.

---

## 패턴 1: 파이프라인 (Pipeline)

**정의:** 각 에이전트가 이전 에이전트의 출력을 입력으로 받아 순차적으로 처리.

**적합한 상황:**
- 단계별 의존성이 명확한 경우
- 각 단계의 출력 형식이 다음 단계의 입력 형식과 일치하는 경우
- 중간 결과물의 품질 검증이 필요한 경우

**에이전트 구성 예시 (교육과정 지식 그래프):**
```
curriculum-parser → standard-extractor → concept-mapper → graph-builder → validator
```

**데이터 흐름:**
```
원문 교육과정 문서
    ↓ (파싱된 JSON)
성취기준 추출
    ↓ (구조화된 성취기준 목록)
개념 관계 매핑
    ↓ (개념-관계 페어)
그래프 구조 생성
    ↓ (완성된 지식 그래프)
검증 및 오류 수정
```

**구현 시 주의사항:**
- 각 단계 사이에 데이터 형식 계약(schema)을 명시한다
- 중간 단계 실패 시 재시작 지점(checkpoint)을 저장한다
- 병목 단계를 식별하고 해당 에이전트에 더 많은 도구를 부여한다

**오케스트레이터 패턴:**
```python
# 파이프라인 실행 의사코드
result = stage1_agent.run(input_data)
if result.success:
    result = stage2_agent.run(result.output)
    ...
else:
    handle_error(result.error, checkpoint="stage1")
```

## 패턴 2: 팬아웃/팬인 (Fan-out/Fan-in)

**정의:** 오케스트레이터가 동일한 작업을 여러 에이전트에게 병렬 분배 후 결과를 수집.

**적합한 상황:**
- 독립적인 작업이 대량으로 존재하는 경우
- 처리 속도가 중요한 경우
- 동일한 작업을 다른 관점/방법으로 처리해야 하는 경우

**에이전트 구성 예시 (코드 리뷰):**
```
              orchestrator
             /      |      \
  arch-reviewer  sec-reviewer  perf-reviewer
             \      |      /
              report-merger
```

**팬아웃 기준:**
- 각 작업이 독립적임 (다른 에이전트 결과에 의존하지 않음)
- 작업당 예상 처리 시간이 비슷함
- 결과 병합 로직이 명확함

**구현 시 주의사항:**
- 각 에이전트가 동일한 출력 형식을 반환하도록 스킬에 명시
- 일부 에이전트 실패 시 부분 결과로 진행할지 전체 재시도할지 결정
- 결과 병합 에이전트에 충돌 해결 로직 포함

---

## 패턴 3: 전문가 풀 (Expert Pool)

**정의:** 오케스트레이터가 작업 특성에 따라 적합한 전문 에이전트를 선택적으로 호출.

**적합한 상황:**
- 입력 유형에 따라 다른 처리 로직이 필요한 경우
- 모든 에이전트를 항상 실행할 필요가 없는 경우
- 전문화된 도메인 지식이 필요한 경우

**에이전트 구성 예시 (음성 채점 시스템):**
```
오케스트레이터 (라우터)
├── phoneme-scorer      ← 발음 평가 요청 시
├── fluency-scorer      ← 유창성 평가 요청 시
├── intonation-scorer   ← 억양 평가 요청 시
├── content-scorer      ← 내용 평가 요청 시
└── comprehensive-scorer ← 종합 평가 요청 시
```

**라우팅 로직 예시:**
```markdown
## 라우팅 규칙

입력 분석:
- task_type == "phoneme" → phoneme-scorer 호출
- task_type == "fluency" → fluency-scorer 호출
- task_type == "comprehensive" → 모든 scorer 병렬 호출 후 집계
- task_type 불명확 → 사용자에게 유형 확인 요청
```

**구현 시 주의사항:**
- 오케스트레이터의 라우팅 로직을 스킬로 분리해 유지보수성 향상
- 새로운 전문가 추가 시 라우팅 스킬만 업데이트하면 되는 구조로 설계
- 각 전문가 에이전트는 자신의 전문 영역 외 요청을 거부하는 명시적 제약 포함

## 패턴 4: 생성-검증 (Producer-Reviewer)

**정의:** 생성 에이전트가 결과물을 만들면 검증 에이전트가 품질을 검수. 기준 미달 시 재생성.

**적합한 상황:**
- 생성 품질이 중요한 경우
- 환각(hallucination) 위험이 있는 AI 생성 콘텐츠
- 특정 형식/표준 준수가 필요한 경우

**에이전트 구성 예시 (AI 문항 은행):**
```
item-generator → hallucination-checker → qti-formatter → quality-reviewer
      ↑                                                          |
      └──────────────── 기준 미달 시 재생성 ◄──────────────────┘
```

**검증 루프 설계:**
```
최대 재시도 횟수: 3회
실패 기준:
  - 환각 감지율 > 5%
  - QTI 형식 오류 존재
  - 난이도 불일치
  - 성취기준 미연결

재생성 시 피드백 전달:
  - 어떤 기준에서 실패했는지
  - 구체적인 오류 위치
  - 수정 방향 제안
```

**구현 시 주의사항:**
- 무한 루프 방지를 위해 최대 재시도 횟수를 명시
- 3회 초과 실패 시 사람의 검토 요청
- 각 재시도마다 피드백을 생성 에이전트에게 전달해 학습 효과 활용

## 패턴 5: 감독자 (Supervisor)

**정의:** 중앙 감독자 에이전트가 전체 흐름을 관리하며 작업을 동적으로 분배. 가장 유연한 패턴.

**적합한 상황:**
- 사전에 정확한 작업 순서를 예측하기 어려운 경우
- 실행 중 결과에 따라 다음 단계가 달라지는 경우
- 여러 패턴이 혼합된 복잡한 워크플로우

**에이전트 구성 예시 (딥 리서치):**
```
supervisor-agent
├── web-searcher       ← 웹 검색 필요 시
├── academic-searcher  ← 학술 자료 필요 시
├── fact-checker       ← 교차 검증 필요 시
├── synthesizer        ← 종합 정리 필요 시
└── report-writer      ← 보고서 작성 시
```

**감독자 의사결정 로직:**
```markdown
## 감독자 판단 기준

현재 상태 평가 후 다음 행동 결정:
1. 수집 정보가 충분한가? → YES: synthesizer 호출, NO: 더 검색
2. 교차 검증이 필요한가? → YES: fact-checker 호출
3. 초안이 완성되었는가? → YES: report-writer 호출
4. 품질 기준을 충족했는가? → NO: 해당 부분 재작업
```

**구현 시 주의사항:**
- 감독자 에이전트의 컨텍스트 창 관리가 핵심 (장기 실행 시 압축 필요)
- 감독자의 판단 로직을 스킬로 명시해 일관성 확보
- 진행 상황을 파일에 저장해 세션 중단 시 재개 가능하게 설계

## 패턴 6: 계층적 위임 (Hierarchical Delegation)

**정의:** 상위 에이전트가 큰 목표를 하위 에이전트에게 작은 목표로 위임. 재귀적 구조.

**적합한 상황:**
- 매우 큰 작업을 계층적으로 분해해야 하는 경우
- 서브태스크가 다시 서브태스크를 생성하는 경우
- 조직 구조가 계층적인 프로젝트 관리

**에이전트 구성 예시 (대규모 소프트웨어 개발):**
```
project-manager
├── frontend-lead
│   ├── ui-developer
│   └── test-engineer
├── backend-lead
│   ├── api-developer
│   └── db-engineer
└── devops-lead
    ├── ci-engineer
    └── monitoring-engineer
```

**위임 프로토콜:**
```markdown
## 위임 시 전달 필수 항목

1. 목표: 이 에이전트가 달성해야 할 것
2. 제약: 하지 말아야 할 것, 범위 한계
3. 형식: 출력 결과의 형식과 저장 위치
4. 기한: 완료 기준 또는 시간 제한
5. 에스컬레이션: 판단하기 어려운 상황에서 상위에게 보고하는 조건
```

**구현 시 주의사항:**
- 각 레이어의 책임 범위를 명확히 정의
- 하위 에이전트의 실패가 상위로 전파되는 방식 명시
- 계층이 3단계를 넘으면 복잡도 급증 — 단순화 검토

---

## 패턴 선택 플로우차트

```
작업이 순차적으로 의존하는가?
├── YES → 파이프라인
└── NO
    │
    ├── 동일 작업을 병렬 처리하는가?
    │   ├── YES → 팬아웃/팬인
    │   └── NO
    │       │
    │       ├── 입력에 따라 다른 전문가가 필요한가?
    │       │   ├── YES → 전문가 풀
    │       │   └── NO
    │       │       │
    │       │       ├── 생성 후 품질 검증이 반복되는가?
    │       │       │   ├── YES → 생성-검증
    │       │       │   └── NO
    │       │       │       │
    │       │       │       ├── 사전에 흐름 예측이 어려운가?
    │       │       │       │   ├── YES → 감독자
    │       │       │       │   └── NO → 계층적 위임
```
